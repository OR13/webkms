Cryptographic authentication systems enable machines, individuals, and organizations to more securely interact with one another. These systems often use public-private key cryptography or encryption mechanisms in order to manage cryptographic material as well as operations utilizing that material. This specification provides a common data model and interface for interacting with these systems enabling one to perform secure cryptographic operations such as keypair creation, wrapping and unwrapping, signing, encrypting, and decrypting.

Introduction

Cryptographic authentication systems enable machines, individuals, and organizations to more securely interact with one another. These systems often use public-private key cryptography or encryption mechanisms in order to manage cryptographic material as well as operations utilizing that material. There are a variety of these sorts of cryptographic systems in use today, sometimes referred to as Key Management Systems. While many of these sorts of systems exist, there is no common interface for accessing and using these systems.

If there were a common API, then you could create drivers for each specific KMS that map to the common API and write applications just once

Drivers could potentially even wrap keystores like FIDO, etc.

If the API was a Web API then you can write Web Apps against this API and servers can offer one or more KMSes -- and potentially even allow users to bring their own

There is a relatively small set of common crypto operations that KMSes provide:

A list is here: https://github.com/digitalbazaar/web-kms-switch/tree/http-sig/lib/schemas

These crypto operations are all really just "methods" that operate on "objects" (keys) in a remote execution environment

There's another spec, called Authorization Capabilities, that is all about remote execution environments with objects and methods -- and granting the authority to remote clients to call those methods on those objects via "capabilities" or "zcaps"

A KMS can provide a Web KMS API that allows users to create keys and execute crypto operations on them by invoking zcaps

The Web KMS API allows users to create keystores (a type of "object") that they control (the user is the "controller" of the keystore)

The keystore has a method for generating a key (another type of "object") of a certain "type" and associating it with that keystore

Each keystore and key is identified by a URI

Each key has a set of "methods", according to its "type", that can be remotely executed via zcaps, such that the key material need never be exfiltrated in order to use the key

In order to execute a method on a key, a KmsOperation of a specific type can be posted to the key URL (using HTTPS), including a zcap invocation via http signatures

A keystore's controller can be changed to enable transfer of keys or to add other parties to help manage the keys
